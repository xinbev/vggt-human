外部论文、论文笔记、论文对应代码仓库或复现参考资料放在 `.paper/` 目录下。该目录主要用于理解论文方法、模型结构、训练策略、损失函数、数据处理和评估逻辑。

主项目代码不得直接从 `.paper/` 中 import 模块。如果参考项目中有可用思想或实现，需要按本项目结构、命名、接口、配置和验证方式重新适配。

## 输出目录

所有生成结果必须放在仓库的 `outputs/` 目录下，并按任务类型创建具体子目录。

Windows 本地输出根目录：

推荐约定：

- 训练输出放在 `outputs/train/`
- 评估输出放在 `outputs/eval/`
- 可视化输出放在 `outputs/vis/`
- 调试或临时分析输出放在 `outputs/debug/`，但不要把无意义临时文件长期留在核心目录

不要把实验结果、日志、图片、导出文件散落在项目根目录或源码目录中。

涉及较复杂的科研改造时，应优先在 `docs/` 中记录：

## 脚本与服务器执行规范

服务器上不允许直接提供或依赖裸 `python ...` 命令执行。因此，凡是需要在服务器上运行的训练、评估、推理、可视化、导出或 smoke test，都应包装成 `.sh` 脚本。

脚本统一放在 `scripts/` 目录下，并按用途细分：

- 训练脚本：`scripts/train/`
- 评估脚本：`scripts/eval/`
- 可视化脚本：`scripts/vis/`
- 工具或 smoke test 脚本：`scripts/tools/` 或 `scripts/smoke/`

## 使用论文与参考代码的规则

使用 `.paper/` 中的论文或代码仓库时，必须先明确：

1. 当前参考的是哪篇论文或哪个项目。
2. 参考的是哪个具体模块、函数、类、训练策略、损失函数或论文思想。
3. 当前实现属于直接移植、适配改写，还是概念重写。
4. 是否保留原 VGGT baseline 路径。
5. 是否需要配置开关控制实验模块。

不要把参考项目中的训练框架、数据集加载器、日志系统、命令行工具或工程脚手架整体搬进主项目，除非用户明确要求。

如果多个参考方向互相冲突，默认优先保留当前 VGGT 行为，除非用户指定采用某个方案。

## 修改 VGGT 的工作流

修改模型结构、训练逻辑、损失函数、回归头、编码器、解码器或评估逻辑之前，必须先检查当前实现。

需要明确：

1. 被修改的是哪个 VGGT 组件。
2. 输入张量和输出张量的 shape。
3. dtype、device、batch 维度和坐标约定。
4. 当前模块在 forward 或训练流程中的位置。
5. 新设计和原始 VGGT 路径的差异。
6. 最小可行集成点。
7. 是否需要保留原实现作为 baseline。

除非任务明确要求整体替换，否则优先新增实验模块、配置项或可选分支，不要直接删除原实现。

## 实验纪律

新增实验时：

1. 不要覆盖 baseline 配置。
2. 实验配置放在 `configs/` 或项目已有配置目录中。
3. 配置命名要能说明研究思路，不要使用 `test`、`new`、`tmp` 这类模糊名称。
4. 临时 debug 代码不要留在核心模型文件中。
5. 重要假设应记录在配置、脚本或 `docs/` 说明文件附近。
6. 大改动应拆成可验证的小步骤。
7. 训练、评估、可视化命令都要整理为 `.sh` 脚本。
8. 所有输出写入 `outputs/` 下的具体子目录。

## 依赖管理

不要轻易引入新依赖。添加依赖前需要先检查：

1. 本项目是否已有等价工具。
2. 是否真的需要完整引入该依赖。
3. 是否会影响 baseline 的可运行性。
4. 是否需要用户在服务器上安装或准备额外资源。

除非集成实现确实需要，否则不要修改依赖文件。

## 验证要求

由于 Windows 本地没有完整执行环境和 ckpt，本地验证通常只能做静态检查、文件结构检查、语法检查或轻量脚本检查。

涉及模型结构改动时，优先准备服务器可运行的最小验证脚本，包括：

1. import 检查。
2. 最小 forward pass。
3. 张量 shape 检查。
4. dtype 和 device 检查。
5. 现有单元测试或 smoke test。
6. 可行时运行最小训练或评估流程。

如果无法在本地运行，需要明确说明：

- 已经完成了哪些静态或结构性验证。
- 需要用户在服务器上运行哪个 `.sh` 脚本。
- 预期输出目录在哪里。
- 尚未验证的风险是什么。

## 沟通要求

完成科研代码改动后，需要总结：

1. 修改了哪个 VGGT 组件。
2. 参考了哪篇论文、哪个项目或哪个思想。
3. 是否保留 baseline。
4. 新增或修改了哪些配置、脚本和文档。
5. 做了哪些本地验证。
6. 需要在服务器上执行哪些脚本。
7. 还存在哪些科研风险或工程风险。
