# Agent 项目指令

## 项目目标

本仓库是一个围绕 VGGT 进行科研改造和实验扩展的代码库。

项目目标是在保留可靠 VGGT baseline 的前提下，逐步实现和验证新的模型结构、训练策略、损失函数、回归头、编码器、解码器和评估逻辑。

优先选择最小、可追踪、可复现的改动。不要为了快速实现而进行大范围重写。

## 主项目代码与参考代码边界

主项目代码位于 `vggt_human/` 目录下，是默认可以执行、导入、测试和修改的代码。

外部论文项目、第三方实现、参考仓库统一放在 `.reference/` 目录下，快速查找参考请阅读`C:\Users\ROG\PycharmProjects\vggt-omega\docs\reference-map.md`

`.reference/` 下的代码默认只读，只用于理解算法、张量流、模型结构、训练方法、损失函数、数据处理和评估逻辑。

除非用户明确要求，不要修改 `.reference/` 下的文件。

主项目代码不得直接从 `.reference/` 中 import 任何模块。

如果参考项目中有有用的思想或实现，需要根据本项目的代码结构、命名规范、接口、配置方式和验证方式，适配到主项目中。

## 使用参考项目的规则

使用 `.reference/` 中的参考项目时，必须先明确：

1. 当前参考的是哪个项目。
2. 当前参考的是哪个具体模块、函数、类或论文思想。
3. 当前实现属于直接移植、适配改写，还是概念重写。
4. 是否保留 VGGT 原始 baseline。
5. 是否需要配置开关控制实验模块。

不要把参考项目中的训练框架、数据集加载器、日志系统、命令行工具、工程脚手架整体搬进主项目，除非用户明确要求。

如果多个参考项目的设计互相冲突，默认优先保留当前 VGGT 行为，除非用户明确指定采用某个方向。

## 改造 VGGT 的工作流

在修改模型结构之前，必须先检查当前 VGGT 的相关实现。

需要明确：

1. 被修改的是哪个 VGGT 组件。
2. 输入张量和输出张量的 shape。
3. dtype、device、batch 维度和坐标约定。
4. 当前模块在 forward 流程中的位置。
5. 新设计和原始 VGGT 路径的差异。
6. 最小可行集成点。
7. 是否需要保留原实现作为 baseline。

除非任务明确要求整体替换，否则优先新增实验模块或配置开关，而不是直接删除原实现。

## 实验纪律

科研实验必须可复现。

新增实验时：

1. 不要覆盖 baseline 配置。
2. 实验配置应放在 `configs/` 或项目已有配置目录中。
3. 配置命名要能说明研究思路，不要使用 `test`、`new`、`tmp` 这类模糊名称。
4. 临时 debug 代码不要留在核心模型文件中。
5. 重要假设应记录在配置、脚本或实验说明附近。
6. 大改动应尽量拆成可验证的小步骤。
7. 训练代码要整理成 shell 脚本启动方式，放置在 `scripts/train/` 目录下，输出到 `outputs/train/`。
8. 可视化脚本放置在 `scripts/vis/` 目录下，输出到 `outputs/vis/`。

## 依赖管理

不要轻易引入参考项目中的新依赖。

添加依赖前需要先检查：

1. 本项目是否已有等价工具。
2. 如需要项目的ckpt或者其他依赖，请说明我会给你处理好
3. 是否真的需要完整引入该依赖。
4. 是否会影响 baseline 的可运行性。

除非集成实现确实需要，否则不要修改依赖文件。

## 验证要求

修改代码后，优先运行最小必要验证。

对于模型结构改动，优先做：

1. import 检查。
2. 最小 forward pass。
3. 张量 shape 检查。
4. dtype 和 device 检查。
5. 现有单元测试。
6. 可行时运行最小训练或评估 smoke test。

如果完整训练或完整评估成本太高，需要明确说明已经验证了什么，尚未验证什么。

## 沟通要求

完成科研代码改动后，需要总结：

1. 修改了哪个 VGGT 组件。
2. 参考了哪个论文项目或哪个思想。
3. 是否保留了 baseline。
4. 做了哪些验证。
5. 还存在哪些科研风险或工程风险。
