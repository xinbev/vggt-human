# Plan: Train SMPL Head To Align With VGGT Scene Point Cloud

## Goal

User only cares that final rendering/visualization has the SMPL human mesh overlap with the VGGT scene point cloud. Therefore this experiment should treat the current VGGT environment point cloud as the target space, even if that space is not metric-correct.

The training objective will be changed from pure metric SMPL supervision to a mixed objective:

- Keep VGGT depth/camera/environment fixed as the visualization coordinate frame.
- Train the SMPL regression head, especially `pred_transl_cam`, to place SMPL mesh/joints into the human-shaped region of the VGGT depth point cloud.
- Keep weak pose/betas/SMPL GT supervision so the body remains plausible and does not collapse to noisy depth blobs.
- Preserve existing baseline configs and training paths.

## Current Findings

Current relevant code:

- `vggt_omega/models/vggt_omega.py`
  - `enable_depth` controls `DenseHead` output `predictions["depth"]`.
  - `enable_camera` controls `pose_enc`.
  - `enable_smpl` controls `pred_poses`, `pred_betas`, `pred_transl_cam`.

- `vggt_omega/models/heads/smpl_head.py`
  - SMPL head predicts `pred_transl_cam` directly.
  - This is currently supervised against BEDLAM `smplx_transl`.

- `vggt_omega/data/bedlam.py`
  - Batch already contains `gt_depth`, `K_scal3r`, `gt_boxes`, `boxes_mask`, `gt_pose_6d`, `gt_betas`, `gt_transl_cam`, `smpl_mask`.
  - `require_depth` can force depth files to exist.

- `vggt_omega/training/hungarian_losses.py`
  - Hungarian loss currently supports SMPL GT, boxes, projected bbox, projected joints.
  - It does not currently train SMPL against VGGT depth point cloud.

- `scripts/vis/visualize_smpl_inference.py`
  - Scene export already puts VGGT depth point cloud and SMPL mesh in the same camera-coordinate visualization if their numerical coordinates match.
  - Environment mesh is built from `predictions["depth"]` and intrinsics decoded from `pose_enc`.
  - Human mesh is built from `SMPL(pred_poses, pred_betas) + pred_transl_cam`.

## Experiment Design

Add a new loss type or extend `HungarianSMPLLoss` with optional scene-alignment terms. Minimal change: extend `HungarianSMPLLoss` with extra weights that default to `0.0`, preserving existing behavior.

New losses:

1. `loss_smpl_depth_z`
   - Project matched SMPL joints or sampled vertices into image coordinates.
   - Sample VGGT predicted depth at those projected pixels.
   - Penalize difference between SMPL point z and VGGT depth z.
   - This directly pulls SMPL translation into the VGGT depth coordinate system.

2. `loss_smpl_point_chamfer`
   - Build a sparse human point cloud from VGGT `depth` inside the matched GT/person bbox.
   - Convert bbox pixels to camera points with the same intrinsics used by visualization.
   - Compare visible/sampled SMPL vertices or joints to those points using robust symmetric Chamfer or one-sided Chamfer.
   - Use small samples to keep memory stable, for example max 256 depth points and 256 SMPL points per matched person.

3. `loss_smpl_silhouette_bbox_depth_optional`
   - Optional and probably not first implementation.
   - Keep projected bbox loss already available; it is enough for first experiment.

Recommended first implementation losses:

```text
L_total =
  existing Hungarian detection/conf/box losses
+ weak L_pose_gt
+ weak L_betas_gt
+ weak L_transl_cam_gt, optional very low or zero
+ scene_depth_z_weight * L_smpl_depth_z
+ scene_chamfer_weight * L_smpl_point_chamfer
+ existing projected_bbox/joints losses if useful
```

For visual-overlap priority:

- `transl_cam_weight` should be reduced from current metric setting, because metric GT translation conflicts with VGGT environment scale.
- `pose_weight` and `betas_weight` should remain nonzero but moderate to keep plausible bodies.
- `joints3d_weight` should be low or disabled if it anchors the body back to BEDLAM metric space.
- Scene alignment losses should mainly update SMPL head, not VGGT depth.

## Intrinsics Choice

For final visualization, environment point cloud currently uses intrinsics decoded from `pose_enc`, not `K_scal3r`. Therefore the scene-alignment loss should use the same intrinsics source as visualization by default:

```yaml
scene_alignment_intrinsics_source: predicted
```

Implementation should also support:

```yaml
scene_alignment_intrinsics_source: gt
```

But first training config should use `predicted` so the loss matches final rendering behavior exactly.

## Gradient Policy

For this visual-alignment experiment:

- Freeze aggregator.
- Freeze camera head.
- Freeze dense depth head.
- Train SMPL head only, plus optional SMPL query tokens / box prior embeddings if needed.

Rationale:

- The VGGT point cloud is the target coordinate space for rendering.
- If dense depth is trainable, the model may move the environment toward SMPL instead of moving SMPL into the existing scene, which is not the requested objective.

Implementation needs a freeze policy extension:

- Existing `apply_freeze_policy` freezes aggregator and camera head.
- Add support for `freeze_dense_head: true`.
- Optionally add `freeze_smpl_pose_shape_heads` later, but first version can train full SMPL head with weak GT regularization.

Add validation:

- `scene_alignment_point_source in {"vertices", "joints"}`.
- Require `predictions["depth"]` when either scene loss weight is nonzero.
- Require `predictions["pose_enc"]` for predicted intrinsics.
- Require `batch["K_scal3r"]` for GT intrinsics.
- Require `smpl_model_dir` because vertices/joints are needed.

Add helper methods:

- `_scene_alignment_losses(...)`
- `_scene_intrinsics(predictions, batch, dtype, device)`
- `_sample_depth_points_from_box(depth, intrinsics, box_cxcywh, image_size, max_points, bbox_expand, depth_conf=None)`
- `_sample_smpl_points(vertices, joints, max_points, source)`
- `_robust_chamfer(points_a, points_b, quantile)`
- `_grid_sample_depth_at_projected_points(depth, projected_uv)`

Implementation details:

- Flatten `depth` from `[B, S, H, W, 1]` to `[B*S, H, W]` if needed.
- Flatten `pose_enc` and `K_scal3r` similarly.
- Use matched `frame_idx`, `src_idx`, and matched target boxes from existing Hungarian matching.
- Build SMPL vertices/joints for matched predictions using existing `_get_smpl_layer`.
- Add `pred_transl_cam` to SMPL points before projection/alignment.
- Detach depth by default if `scene_alignment_detach_depth=true`, so gradients update SMPL only.
- For depth point cloud, use matched target bbox as rough human mask because current dataset has boxes but no segmentation mask.
- Sample regular grid pixels inside bbox rather than all pixels for efficiency.
- Filter depth points with `z > 1e-6` and optional `depth_conf` threshold.
- Use robust quantile trimming so background pixels inside bbox do not dominate.

Loss behavior:

- `loss_scene_depth_z`: one-sided loss from SMPL projected points to depth map.
- `loss_scene_chamfer`: one-sided or symmetric robust Chamfer between translated SMPL points and bbox depth points.
- Return zero tensors when no valid matches/points.
- Add metrics:

```python
metric_scene_depth_z_l1
metric_scene_chamfer
metric_scene_valid_pairs
```

Add to total:

```python
+ self.scene_depth_z_weight * losses["loss_scene_depth_z"]
+ self.scene_chamfer_weight * losses["loss_scene_chamfer"]
```

## Risks And Mitigations

Risk: bbox contains background depth, causing SMPL to align to background planes.

Mitigation:

- Use robust quantile trimmed Chamfer.
- Keep `scene_chamfer_weight` moderate.
- Use `scene_depth_z_weight` first, then increase Chamfer if needed.

Risk: SMPL shape gets distorted by noisy depth.

Mitigation:

- Keep `pose_weight` and `betas_weight` nonzero.
- Consider freezing pose/betas heads later and training only `transl_cam_heads` if distortion appears.

Risk: predicted depth and predicted intrinsics are both from baseline and may not match GT boxes perfectly.

Mitigation:

- This is acceptable because final visualization uses predicted depth/intrinsics.
- If alignment is unstable, switch `scene_alignment_intrinsics_source: gt` for debugging.

Risk: memory from Chamfer distance.

Mitigation:

- Sample at most 256 SMPL points and 256 depth points per matched person.
- Use `torch.cdist` only on sampled points.

## Expected Outcome

After this implementation and training, the SMPL head will learn to output bodies in the same coordinate space as the VGGT scene point cloud used by rendering. This is explicitly optimized for final visual overlap, not for preserving BEDLAM metric correctness.
